# خطة اختبارات — منصة سند
**الهدف:** تغطية اختبارية حقيقية للمناطق الحرجة، بحيث نقفل باب "مفيش تست" نهائيًا زي ما اتقفل باب `any`.
**نقطة البداية:** 4 ملفات تست فقط (~300 سطر) بتغطي دوال utility معزولة. مفيش تغطية على الـ hooks، الـ Edge Functions، أو عزل الـ tenants.

**مرجع الاستعلامات:** `docs/reference/مرجع-فحص-Supabase-عزل-tenants.md` — كل استعلام SQL اتعمل فعليًا على Supabase (schema، RLS policies، دوال مساعدة، أعمدة إجبارية، توزيع بيانات الإنتاج) مع نتيجته الحقيقية، خاص بتجهيز مرحلة 1. أي حاجة جديدة (اسم جدول/عمود/دالة) لازم تتأكد بنفس الأسلوب — استعلام فعلي، مش تخمين — وتتضاف هناك.

## منهجية العمل (زي خطة تنظيف `any` بالظبط)

- ملف/وحدة واحدة في كل مرة، مش كل حاجة مرة واحدة.
- كل دفعة بتتسلم، تتشغّل عندك (`npm run test`)، وتتأكد إنها فعلاً بتفشل لو كسرنا المنطق (مش تست وهمي بينجح دايمًا).
- مفيش تعديل على منطق العمل نفسه أثناء كتابة التست — لو التست كشف باگ، بيتوثق في تقرير منفصل زي أعطال البيانات اللي اتكشفت بالأنواع، وميتصلحش جوه نفس الدفعة إلا لو اتفقنا.
- بعد كل مرحلة، بنحدّث الجدول في آخر الملف ده (✅ / 🟡 / 🔴) بدل ما نعمل تقرير جديد كل مرة.

### ملاحظة مهمة: نوعين من الاختبارات مختلفين
1. **اختبارات منطق (Vitest)** — نفس أسلوب `dataAccess.test.ts` الموجود: بنعمل mock لسلسلة استدعاءات Supabase (`db.from().select().eq()`...) ونتأكد إن الدالة بترجع/بتكتب الصح. ده بيغطي الـ hooks والـ Edge Functions logic.
2. **اختبارات عزل قاعدة البيانات (RLS)** — دي **مش قابلة للاختبار بالـ mock**، لأن الموضوع كله إن قاعدة البيانات نفسها (مش الكود) هي اللي بترفض الطلب. محتاجة سكريبت SQL بيتشغل على مشروع Supabase تجريبي (staging) فيه tenant تجريبيين، ويتأكد فعليًا إن tenant A مينفعش يقرا/يكتب بيانات tenant B. المرحلة 1 تحت بتوضح الأسلوب ده بالتفصيل.

## المرحلة 1 — عزل الـ Tenants (RLS) 🔴 الأولوية القصوى

ده أخطر جزء ينكسر من غير ما حد ياخد باله، لأنه بيحصل على مستوى قاعدة البيانات مش الواجهة.

**الأسلوب:** سكريبت SQL (أو مجموعة سكريبتات) بيتشغل يدويًا على مشروع staging منفصل عن الإنتاج، بينشئ:
- تينانت تجريبي A وتينانت تجريبي B، كل واحد بمستخدم وبيانات (قضية، موكل، جلسة، فاتورة) خاصة بيه.
- محاولات قراءة/كتابة/حذف متقاطعة (مستخدم A بيحاول يقرا بيانات B) على **كل جدول فيه `tenant_id`**، ويتأكد إن كل محاولة اترفضت.

**الجداول المستهدفة (لازم تتغطى كلها، مش عينة):**
| المجموعة | الجداول |
|---|---|
| القضايا والجلسات | `cases`, `case_sessions`, `case_documents` |
| الموكلين | `clients` |
| الفلوس | `case_fees`, `fee_payments`, `invoices` |
| التذكيرات والنشاط | `reminders`, `activity_log` |
| الإدارة | `profiles`, `office_settings` (لو فيها بيانات لكل تينانت) |
| المكتبة القانونية / AI | أي جدول بيتخزن فيه محتوى خاص بمكتب معيّن |

**حالات إضافية غير عزل التينانت:**
- محاولة تصعيد صلاحيات (مستخدم عادي بيحاول يعدّل `role` بتاعه لـ admin) — تأكيد إن الـ trigger اللي اتعمل للـ profile escalation لسه شغال.
- مستخدم `is_active = false` — يتأكد إنه مرفوض من كل نقطة، مش بس تسجيل الدخول.

## المرحلة 2 — الـ Edge Functions 🔴 أولوية قصوى

دي بوابات بتستخدم `service_role` (صلاحيات كاملة)، فأي ثغرة فيها = تجاوز لكل الـ RLS اللي بنينا.

**كل فانكشن محتاج تست لمنطق `getAuthorizedCaller` بتاعه (من `_shared/auth.ts`) بالحالات دي كحد أدنى:**
- من غير `Authorization` header → 401
- توكن منتهي/غلط → 401
- حساب موجود بس `is_active: false` → 403
- حساب من غير الصلاحية المطلوبة (مثلاً موظف عادي بيحاول يستخدم فانكشن أدمن) → 403
- الطلب سليم → بيكمل وبيرجع النتيجة المتوقعة

| الفانكشن | إيه اللي محتاج اختبار خصوصي غير الـ auth |
|---|---|
| `saas-admin` | عمليات على مستوى الـ SaaS كله — تأكيد إنها super-admin بس |
| `admin-actions` | كل action بيتاخد فيه decision بناءً على `role` — كل حالة role لوحدها |
| `client-portal-api` | ده بوابة للعميل الخارجي (مش موظف المكتب) — لازم تست إضافي إن عميل من مكتب معيّن مينفعش يشوف بيانات مكتب تاني عن طريق البوابة دي |
| `office-login` / `office-secrets` | تسجيل الدخول والأسرار — تأكيد عدم تسريب معلومات في رسائل الخطأ (مثلاً فرق بين "اليوزر مش موجود" و"الباسورد غلط" ممكن يبقى تسريب) |
| `ai-chat` / `embed-batch` / `embed-query` / `process-law-extract` | تأكيد إن السياق (context) اللي بيتبعت للموديل الخارجي محصور في بيانات تينانت الطالب بس |
| `session-alerts` / `telegram-send` | تأكيد إن الرسالة بتتبعت لتينانت/رقم الشخص الصح فقط |

**السبب الأول (الستة تستات في `البحث`) — مؤكد ومصلّح:**
`vi.useFakeTimers()` كان متفعّل في `beforeEach` بتاع الـ describe، يعني قبل ما `renderReady()` تتنفذ جوه كل تست. `renderReady()` بتستخدم `waitFor()` من `@testing-library/react`، واللي بيعتمد داخليًا على `setTimeout`/`setInterval` حقيقيين عشان يعمل polling على شرط `loading=false`. لما الـ fake timers شغالة من الأول، الـ polling ده مبيتقدمش خالص (محتاج `advanceTimersByTime` يدوي) → الانتظار بيفضل معلّق لحد ما يضرب الـ testTimeout الافتراضي (5000ms).
**التصحيح:** نقل `vi.useFakeTimers()` من الـ `beforeEach` لداخل كل تست، بعد `renderReady()` مباشرة (يعني بعد ما التحميل الأولي يخلص بالـ timers الحقيقية، وبعدين نفعّل fake timers بس لحظة اختبار الـ debounce). الستة تستات اتصلحوا (اتأكد بتشغيل فعلي تاني: نزلوا من الفشل).

**السبب التاني (تست `tenant_id` في `handleSave`) — مؤكد ومصلّح:**
التست كان بيكتب `{ ...profile, tenant_id: null }` *جوه* الكولباك اللي بتتبعت لـ `renderHook` مباشرة — يعني object جديد بـ identity مختلفة في كل re-render. الهوك حاطط `profile` في dependency array بتاع `useEffect(fetchReminders)`، فـ identity المتغيّرة دي بتخلي الـ effect يشتغل تاني كل re-render → `fetchReminders()` → `setLoading(true)` → re-render → object جديد تاني → loop عمليًا مفيهوش نهاية → 5000ms timeout. الـ loop المعلّق ده محتمل يكون هو اللي "سحب" معاه تست الـ INSERT الناجح اللي بعده (الـ event loop بيفضل مشغول بيه).
**التصحيح:** تثبيت الـ object في متغيّر (`profileNoTenant`) *برّه* الكولباك عشان يفضل نفس الـ reference بين الرندرات.

**الحالة بعد التصحيحين (لسه محتاج تأكيد بتشغيل فعلي):** المتوقع نزول العدد لـ **3 فشل أو أقل** (تستين handleSave المصلّحين + احتمال تست الـ INSERT الفاشل يتصلح تلقائيًا لو كان أثر جانبي من الـ loop). **لو لسه فيه فشل بعد التشغيل الجديد، مطلوب نص الخطأ الكامل لكل تست (مش بس العنوان) عشان نحدد أي سبب مستقل تاني بدل التخمين.**
