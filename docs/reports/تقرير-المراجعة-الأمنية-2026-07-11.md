## ملخص تنفيذي

تمت مراجعة أمنية شاملة لكل مكونات المنصة على Supabase. تم اكتشاف **9 مشاكل أمنية** بدرجات خطورة متفاوتة، من بينها **2 ثغرتين حرجتين (Critical)** كانتا تسمحان بتصعيد صلاحيات كامل على المنصة والوصول الكامل لقاعدة البيانات. تم إصلاح **جميع المشاكل المكتشفة** خلال جلسة المراجعة نفسها، بالإضافة إلى تحديث كود الفرونت إند ليتوافق مع التغييرات الأمنية الجديدة.

## 2. 🔴 ثغرة search_path في دوال SECURITY DEFINER

**الوصف:** دوال قاعدة البيانات التي تعمل بصلاحيات SECURITY DEFINER (تُنفَّذ بصلاحيات منشئها بدل المستخدم المستدعي) بدون تثبيت `search_path` صراحةً تكون عرضة لهجوم **search_path hijacking**: يمكن لمستخدم خبيث — نظريًا عبر جدول مؤقت (temp table) بنفس اسم الجدول الحقيقي — خداع الدالة لتقرأ من جدول مزيّف بدل الجدول الأصلي.

**الدوال المتأثرة (5):**
- `get_my_role()`
- `get_overdue_sessions()`
- `log_activity()`
- `notify_telegram_message()`
- `verify_client_pin(text, text)` — لاحقًا تبيّن أنها دالة غير مستخدمة (انظر البند 7) وتم حذفها بالكامل

**الإصلاح:**
```sql
ALTER FUNCTION public.get_my_role() SET search_path TO 'public';
ALTER FUNCTION public.get_overdue_sessions() SET search_path TO 'public';
ALTER FUNCTION public.log_activity() SET search_path TO 'public';
ALTER FUNCTION public.notify_telegram_message() SET search_path TO 'public';
```

**الحالة:** ✅ تم التنفيذ والتحقق.

## 3. 🔴🔴 Storage Buckets عامة بالكامل (بيانات عملاء حقيقية)

**الوصف:** باكتات `case-docs` (مستندات القضايا) و`client-docs` (صور البطاقة الشخصية وعقود التوكيل للعملاء) كانت مضبوطة `public = true`. أي شخص يعرف رابط الملف — من غير أي تسجيل دخول — كان يقدر يفتح/يحمّل الملف مباشرة، متجاوزًا أي RLS على مستوى الجداول (RLS الجداول لا يحمي كائنات Storage).

**دليل الخطورة:** أسماء الملفات كانت تتبع نمط `prefix_<timestamp بالميلي ثانية>` (مثل `id_1780785145413.jpg`, `poa_1780785146732.jpg`) — وهو نمط قابل للتخمين ضمن نطاق زمني محدود، خصوصًا مع توفر أوقات الأنشطة في جدول `activity_log`.

**الإصلاح:**
1. نقل الملفات "التائهة" (خارج فولدر الـ tenant) إلى مسارها الصحيح يدويًا.
2. إغلاق الباكتات:
   ```sql
   update storage.buckets set public = false where id in ('case-docs', 'client-docs');
   ```
3. **تحديث الكود بالكامل** للانتقال من الروابط العامة الثابتة إلى **روابط موقّعة مؤقتة (Signed URLs)**:
   - إضافة دوال مساعدة (`getSignedUrl`, `resolveStorageUrl`, `useResolvedStorageUrl`) في `utils.ts`.
   - تحديث كل نقاط الرفع والعرض: `ArchiveTab.tsx`, `useCaseDetailActions.ts`, `useClientActions.ts`, `useAdminOffice.ts`, `ClientDetailModal.tsx`, `EditClientModal.tsx`, `constants.ts`.
   - تحديث Edge Function `client-portal-api` لتوليد روابط موقّعة سيرفر-سايد بدل إرجاع الرابط العام القديم.

**الحالة:** ✅ تم التنفيذ والتحقق (تأكيد عدم وجود ملفات خارج فولدرات الـ tenant).

## 4. 🔴 مكتبة القوانين مفتوحة لأي زائر (Anonymous)

**الوصف:** جداول `law_articles`, `laws`, `legal_categories` كانت متاحة للقراءة لأي شخص (`using: true`, role: `public`) بدون أي تسجيل دخول. بالإضافة لذلك، دالة `search_law_articles` (SECURITY DEFINER، تتخطى RLS بالكامل) كانت ممنوحة صلاحية `EXECUTE` لـ `anon`.

**القرار (بناءً على رغبة صاحب المشروع):** المكتبة لسه قيد الإعداد ومفروض تكون مقفولة تمامًا حاليًا إلا لصاحب النظام (super admin)، مع إبقاء وصول "المساعد القانوني الذكي" (AI Legal Assistant) شغال للمحامين المسجلين (authenticated) فقط.

**الإصلاح:**
```sql
drop policy if exists "Allow read law_articles" on law_articles;
drop policy if exists "Admin manage law_articles" on law_articles;
drop policy if exists "Allow read laws" on laws;
drop policy if exists "Admin manage laws" on laws;
drop policy if exists "Allow read categories" on legal_categories;

create policy "super_admin_only_law_articles" on law_articles
  for all using (is_super_admin()) with check (is_super_admin());
create policy "super_admin_only_laws" on laws
  for all using (is_super_admin()) with check (is_super_admin());
create policy "super_admin_only_legal_categories" on legal_categories
  for all using (is_super_admin()) with check (is_super_admin());

revoke execute on function public.search_law_articles(text, integer, uuid[]) from anon;
revoke execute on function public.match_law_articles(vector, integer, uuid) from anon, authenticated;
```

**الحالة:** ✅ تم التنفيذ.

## 8. 🔴🔴 تصعيد صلاحيات عبر إدراج صفوف profiles (الأخطر في المراجعة)

**الوصف:** الـ `WITH CHECK` الخاص بسياسة الإضافة (`profiles_insert`) على جدول `profiles` كان يتحقق فقط من أن `tenant_id` المُدخل يطابق مكتب المستخدم الحالي، وأن دوره `admin` — لكنه **لم يكن يتحقق إطلاقًا من قيم `role` أو `is_super_admin`** في الصف الجديد نفسه. هذا يعني أن أي أدمن مكتب عادي (غير super admin) كان يستطيع إدراج صف `profiles` جديد لنفسه بقيمة `is_super_admin = true`، مانحًا نفسه صلاحيات كاملة على المنصة بأسرها.

**مُضاعِف الخطورة:** عمود `user_id` في `profiles` لم يكن عليه قيد `UNIQUE`، مما يسمح بوجود أكثر من صف لنفس المستخدم (صف "شبح" إضافي بصلاحيات مرتفعة، من غير المساس بالصف الأصلي).

**الإصلاح:**
```sql
-- منع تكرار صف profile لنفس المستخدم نهائيًا
alter table public.profiles
  add constraint profiles_user_id_key unique (user_id);

-- قفل الإضافة المباشرة على super admin فقط
-- (كل إنشاء حسابات فعلي يمر عبر admin-actions بصلاحية service_role،
--  فلا يوجد استخدام شرعي لهذه الـ policy)
drop policy if exists "profiles_insert" on profiles;
create policy "profiles_insert_super_admin_only" on profiles
  for insert
  with check (is_super_admin());
```

**ملاحظة إيجابية:** تم التحقق من وجود trigger (`protect_profile_privileges`) يمنع بالفعل أي مستخدم من تعديل (`UPDATE`) الأعمدة الحساسة (`role`, `is_super_admin`, `tenant_id`, `permissions`, `is_active`, `is_locked`) في صفه الخاص — المشكلة كانت محصورة في مسار الإضافة (`INSERT`) فقط.

**الحالة:** ✅ تم الإصلاح والتحقق.

## 9. 🔴🔴 مفتاح service_role مكشوف نصًا صريحًا في cron.job

**الوصف:** أثناء فحص المهام المجدولة (pg_cron)، تبيّن أن مفتاح `service_role` الكامل (يتخطى RLS بالكامل، صلاحية تصل حتى عام 2096) بالإضافة إلى سر التحقق (`x-cron-secret`) الخاص بفانكشن `session-alerts`، كانا **مكتوبين نصًا صريحًا** داخل عمود `command` في جدول `cron.job`. هذا الجدول قابل للقراءة من أي جهة لديها صلاحية اطلاع على schema الـ `cron`.

**الإصلاح:** نقل السرّين إلى **Supabase Vault** (مشفّرين)، وتعديل أوامر الـ cron jobs (`morning-alerts`, `evening-alerts`) لتسحب القيم من Vault ديناميكيًا وقت التنفيذ بدل تضمينها كنص ثابت:
```sql
select vault.create_secret('<service_role_key>', 'cron_service_role_key');
select vault.create_secret('<cron_secret>', 'session_alerts_cron_secret_value');

select cron.alter_job(job_id := 12, command := $cmd$
  select net.http_post(
    url := 'https://<project-ref>.supabase.co/functions/v1/session-alerts',
    headers := jsonb_build_object(
      'Content-Type', 'application/json',
      'Authorization', 'Bearer ' || (select decrypted_secret from vault.decrypted_secrets where name = 'cron_service_role_key'),
      'x-cron-secret', (select decrypted_secret from vault.decrypted_secrets where name = 'session_alerts_cron_secret_value')
    ),
    body := '{"type":"morning"}'::jsonb
  );
$cmd$);
-- ونفس التعديل لـ job_id := 9 (evening-alerts)
```

**⚠️ توصية لم تُنفَّذ بعد (قرار صاحب المشروع):**
تم التوصية بشدة بتدوير (Rotate) مفتاح `service_role` نفسه من Supabase Dashboard → Project Settings → API، لأن قيمته أصبحت مكشوفة ليس فقط في قاعدة البيانات بل أيضًا داخل سجل هذه المحادثة. صاحب المشروع اختار عدم التدوير حاليًا وتفضيل تأمين آلية التخزين فقط. **هذا قرار مفتوح ينصح بمراجعته لاحقًا.**

**الحالة:** ✅ آلية التخزين تم تأمينها بالكامل. ⏳ تدوير المفتاح نفسه — قرار مؤجل.

## 11. مراجعة صحة الكود الفعلي المنشور (Edge Functions) — جلسة متابعة

بعد إتمام المراجعة الأمنية، تم اقتراح مهم من صاحب المشروع: **التأكد من الكود الحقيقي المنشور فعليًا على Supabase**، لأن نسخة الملفات المحلية (Zip) قد لا تطابق ما هو منشور فعليًا بعد تعديلات لاحقة. تم سحب كود كل الـ 10 Edge Functions مباشرة من Supabase Dashboard ومقارنتها حرفيًا بالنسخة المحلية.

### النتيجة: هذا الافتراض كان صحيحًا ومهمًا — تم اكتشاف اختلافات جوهرية

| الفانكشن | النتيجة |
|---|---|
| `admin-actions`, `office-secrets`, `session-alerts`, `office-login`, `telegram-send`, `ai-chat` | ✅ مطابقة تمامًا للنسخة المحلية |
| `embed-batch`, `embed-query` | ✅ مطابقة منطقيًا (فروق شكلية فقط في طريقة التجميع) |
| `saas-admin` | 🔴 **فرق حقيقي** |
| `process-law-extract` | 🟡 فرق بسيط |
| `client-portal-api` | 🔴🔴 **فرق جوهري خطير** |

### 11.1 🔴🔴 `client-portal-api`: بوابة العملاء كانت عاطلة فعليًا في الإنتاج

اتضح أن النسخة المنشورة فعليًا (المستخدمة من قِبل العملاء الحقيقيين عبر `client-portal.html`) كانت تستعلم عن **جداول غير موجودة أصلاً** في قاعدة البيانات، نتيجة عدم تحديث الفانكشن بعد إعادة تسمية الجداول في مرحلة سابقة من المشروع:

| الأكشن | الجدول الخاطئ (منشور) | الجدول الصحيح |
|---|---|---|
| `getCaseFees` | `fees` (غير موجود) | `case_fees` |
| `getCaseSessions` | `sessions` (غير موجود) | `case_sessions` |
| `getCaseDocuments` | `documents` (غير موجود) | `case_documents` |
| `getMessages` / `sendMessage` | `portal_messages` (غير موجود) | `client_messages` |

بالإضافة إلى ذلك، كان الاستعلام عن `cases` يطلب عمود `client_name` غير الموجود في الجدول أصلاً.

**تم أيضًا أثناء هذا الإصلاح:**
- تحديث توليد روابط المستندات لتصبح روابط موقّعة (Signed URLs) بدلًا من الروابط العامة القديمة، بما يتوافق مع إغلاق باكت `case-docs` (بند 3).
- إضافة فحص حالة اشتراك المكتب (`tenant.status`) عند تسجيل دخول العميل للبوابة، بنفس النمط المُطبَّق مسبقًا في `office-login` — لمنع عملاء مكتب موقوف الاشتراك من الاستمرار في الوصول لبوابتهم.

**ملاحظة منهجية مهمة:** النسخة المحلية (Zip) كانت هي الأخرى تحتوي أخطاءً مختلفة في نفس الاستعلامات (أسماء أعمدة خاطئة مثل `court:court_name`)، أي أن كلا النسختين (المحلية والمنشورة) كانتا خاطئتين بطرق مختلفة. تم التحقق من الأعمدة الحقيقية مباشرة من قاعدة البيانات (`information_schema.columns`) بدلًا من الاعتماد على أي من النسختين، لضمان دقة الإصلاح النهائي.

**الحالة:** ✅ تم إعادة كتابة الأكشنز الأربعة بالكامل بالأسماء الصحيحة المؤكدة من قاعدة البيانات، وتم النشر.

### 11.2 🔴 `saas-admin`: ميزة فواتير المكاتب كانت مقفولة في الإنتاج

```ts
// المنشور فعليًا:
const ALLOWED_TABLES = ['tenants'];
// الصحيح (تستخدمه لوحة offices-portal.html فعليًا لعرض/إضافة/حذف الفواتير):

```
**الحالة:** ✅ تم الإصلاح والنشر.

### 11.3 🟡 `process-law-extract`: تحقق ناقص من نجاح عملية الحذف
عملية حذف مواد القانون القديمة قبل إضافة المواد الجديدة (عند إعادة معالجة قانون) لم تكن تتحقق من نجاحها، مما قد يؤدي نظريًا لتكرار/خلط بيانات لو فشل الحذف لأي سبب.
**الحالة:** ✅ تم إضافة `if (delErr) throw delErr;`.

### 11.4 🟡 `admin-actions`: باگ كامن (غير قابل للاستغلال حاليًا عبر الواجهة)
دالة `create_lawyer` كانت تسند `tenant_id` للحساب الجديد من `caller.tenant_id` مباشرة. لو نُفِّذت من سوبر أدمن بلا مكتب مرتبط بحسابه (`tenant_id = null`)، كان الحساب الجديد سيُنشأ بـ `tenant_id = null` (حساب معطوب). تم التحقق من أن كل نقاط الاستدعاء الحالية في الواجهة تخص أدمن مكتب حقيقي فقط (المشكلة غير قابلة للحدوث حاليًا)، لكن تم إصلاحها للتحصين المستقبلي بإضافة دعم صريح لتحديد `target_tenant_id` عند استخدام سوبر أدمن للإجراء.
**الحالة:** ✅ تم الإصلاح والنشر.

## 12. جولة متابعة أخيرة: صلاحيات دوال SECURITY DEFINER + إعدادات البنية التحتية

### 12.1 🔴 `get_overdue_sessions()`: تسريب بيانات قضايا كل المكاتب

دالة SECURITY DEFINER بترجع الجلسات المتأخرة (بعنوان القضية، اسم المحكمة، اسم المدعي والمدعى عليه) **بدون أي فلترة بـ `tenant_id`**، وكانت ممنوحة `EXECUTE` لـ `anon` و`authenticated`. أي مستخدم مسجّل — أو حتى زائر غير مسجل دخول — كان يقدر ينادي عليها مباشرة (`db.rpc('get_overdue_sessions')`) ويحصل على بيانات قضايا حساسة من **كل المكاتب على المنصة**، وليس مكتبه فقط. تم التأكد من عدم استخدامها في أي مكان بالكود الحالي (كود ميت).

**الإصلاح:**
```sql
revoke execute on function public.get_overdue_sessions() from anon, authenticated, public;
```
**الحالة:** ✅ تم الإصلاح.

### 12.2 🟡 `generate_invoice_number(p_tenant_id uuid)`: تلاعب بعداد فواتير مكتب آخر

الدالة كانت تقبل `p_tenant_id` كوسيط دون التحقق من انتماء المستخدم المستدعي لهذا المكتب، مما يسمح لمستخدم في مكتب A بزيادة عداد الفواتير (`invoice_counter`) الخاص بمكتب B عبر استدعاء الدالة مباشرة بمعرّف مكتب مختلف عن مكتبه.

**الإصلاح:** إضافة تحقق داخل الدالة:
```sql
IF p_tenant_id IS DISTINCT FROM current_tenant_id() AND NOT is_super_admin() THEN
    RAISE EXCEPTION 'غير مصرح بإنشاء رقم فاتورة لمكتب آخر';
END IF;
```
**الحالة:** ✅ تم الإصلاح.

### 12.3 🟢 تنظيف إضافي: `log_activity()` و`notify_telegram_message()`

دالتان من نوع trigger (`RETURNS trigger`) لا يُفترض استدعاؤهما مباشرة أصلًا، لكنهما كانتا تحملان `EXECUTE` لـ `anon`/`authenticated`/`PUBLIC` بلا داعٍ عملي (استدعاؤهما المباشر يفشل فورًا لعدم توفر سياق trigger، فلم تكونا ثغرة فعلية، لكن أُزيلت الصلاحية كتحصين إضافي وتقليل سطح الهجوم).
```sql
revoke execute on function public.log_activity() from anon, authenticated, public;
revoke execute on function public.notify_telegram_message() from anon, authenticated, public;
```
**الحالة:** ✅ تم التنفيذ.

### 12.4 مراجعة دوال الأسرار الحساسة (Groq/Telegram)

تم التحقق من أن كل الدوال التي تتعامل مع مفاتيح API الحساسة (`get_office_groq_key`, `set_office_groq_key`, `get_office_tg_daily_token`, `set_office_tg_daily_token`, `get_office_tg_instant_token`, `set_office_tg_instant_token`, `admin_force_logout`, `get_all_daily_tg_configs`) **غير ممنوحة لأي دور غير موثوق** (`anon`/`authenticated`) من الأساس — لا يمكن استدعاؤها إلا عبر Edge Functions بصلاحية `service_role`. ✅ سليمة، لا حاجة لأي إجراء.

### 12.5 إعدادات البنية التحتية (Infrastructure)

| البند | الحالة الحالية | التوصية |
|---|---|---|
| **Database Backups** | المشروع على **الخطة المجانية** — لا يوجد نسخ احتياطي تلقائي إطلاقًا (لا يومي ولا Point-in-Time) | ⚠️ يجب الترقية لخطة Pro قبل استقبال بيانات عملاء حقيقيين — فقدان البيانات في هذه الحالة لا رجعة فيه |
| **Network Restrictions** | الاتصال المباشر بقاعدة البيانات (Postgres) متاح من أي IP في العالم | مقبول حاليًا (التطبيق يتواصل عبر REST API المحمي بـ RLS وليس اتصالًا مباشرًا)؛ يُنصح بالتقييد لاحقًا عند توفر IP ثابت (مكتب/سيرفر CI) |
| **Auth Rate Limits** | القيم الافتراضية لـ Supabase (مثلًا 30 محاولة تسجيل دخول/5 دقائق لكل IP) | ✅ قيم معقولة، لا تغيير مطلوب حاليًا |

## 13. ملاحظة منهجية مهمة: Default Privileges على public schema

تم التحقق من `pg_default_acl` وتأكد أن **أي جدول أو دالة أو sequence جديدة تُنشأ في `public` schema تأخذ تلقائيًا صلاحيات كاملة لـ `anon` و`authenticated`** (بما فيها `EXECUTE` على أي دالة جديدة) — هذا سلوك افتراضي قياسي في Supabase، وليس خطأ إعداد، حيث الاعتماد الأساسي للحماية يكون على RLS وليس على الـ Grants نفسها.

هذا يفسر لماذا تكرر نفس نمط الثغرة (`search_law_articles`, `get_overdue_sessions`) أكثر من مرة في هذه المراجعة: أي دالة `SECURITY DEFINER` جديدة تُنشأ مستقبلًا ستكون **قابلة للاستدعاء المباشر من أي مستخدم عادي تلقائيًا**، حتى دون قصد.

### ✅ Checklist لأي دالة SECURITY DEFINER جديدة مستقبلًا
عند إنشاء أي دالة جديدة بـ `SECURITY DEFINER`، يجب في نفس الـ migration:
1. تثبيت `SET search_path TO 'public'` صراحةً (لمنع hijacking).
2. إذا لم تكن الدالة مخصصة للاستدعاء المباشر من الواجهة (مثلًا: دالة مساعدة لسياسات RLS فقط، أو دالة يُفترض استدعاؤها فقط عبر Edge Function بصلاحية service_role)، أضف صراحةً:
   ```sql
   revoke execute on function public.<function_name>(<args>) from anon, authenticated, public;
   ```
3. إذا كانت الدالة تقبل معرّف `tenant_id`/`client_id` كوسيط، تأكد من وجود تحقق داخلي أن هذا المعرّف يطابق `current_tenant_id()` الخاص بالمستدعي (أو `is_super_admin()`)، وليس الاعتماد فقط على أن "المفروض" لا أحد يستدعيها بقيمة خاطئة.

## 14. تحصين إضافي: التحقق من شكل المعرّفات (UUID) قبل بناء استعلامات PostgREST

لوحظ أن بعض الفانكشنز تبني روابط استعلام PostgREST عبر دمج نصي مباشر لقيم قادمة من طلب المستخدم (مثل `` `cases?id=eq.${caseId}` ``) بدلًا من استخدام مكتبة استعلامات تقوم بالـ escaping تلقائيًا. تم فحص هذا النمط عبر كل الفانكشنز:

- **`client-portal-api` (`caseId`)**: القيمة قادمة مباشرة من جسم الطلب دون أي تحقق من شكلها. التقييم: الخطورة الفعلية منخفضة (PostgREST يجمع كل شروط الاستعلام بـ `AND` افتراضيًا، وفحص الملكية يتم عبر استعلام تحقق منفصل قبل أي عملية تالية)، لكن الاعتماد على تفاصيل تنفيذ PostgREST الداخلية كخط دفاع وحيد ممارسة غير آمنة. **تم إضافة تحقق صريح من شكل UUID** (`isValidUuid`) قبل استخدام القيمة في أي استعلام.
- **`admin-actions` (`user_id`)**: محمية بالفعل بشكل غير مباشر عبر دالة `authorizeOnTarget` التي تتحقق من ملكية المستخدم الهدف لنفس مكتب الأدمن المستدعي قبل تنفيذ أي عملية حساسة؛ قيمة غير صالحة الشكل سترسب في هذا التحقق أولًا. لا حاجة لتعديل إضافي.

**الحالة:** ✅ تم إضافة التحصين في `client-portal-api` ونشره.

7. تنظيف اختياري: نقل ملفات شعار المكتب القديمة (4 ملفات في `client-docs/office/`) لفولدر الـ tenant الموحّد، أو حذفها.
